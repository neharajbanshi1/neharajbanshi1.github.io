Background

A dataset hiding in plain sight

I watch a lot of films. So does my friend Nirjan. We argue about them constantly.

At some point it occurred to me that we were sitting on a dataset — both of us had been logging watches, ratings, and reviews on Letterboxd for years. So instead of debating who has better taste, I decided to actually look at the data.

I exported both our Letterboxd diaries as CSVs, imported them into a shared PostgreSQL database, and wrote queries to compare our watching habits, rating patterns, and biggest points of disagreement. What started as SQL practice turned into something I genuinely wanted to explore.

The problem

Real data is never clean

Letterboxd's CSV export is designed for one person. To do a meaningful cross-person analysis, several real problems had to be solved before writing a single interesting query.

⚠️

Schema mismatch

My export used capitalised column names ("Name", "Date") while Nirjan's used lowercase. Every query touching both tables needed careful aliasing.

🧹

Dirty data

Rewatches created duplicates. Missing ratings made numerical comparison impossible. Inconsistent film titles caused joins to silently miss matches.

🔑

No shared key

Letterboxd doesn't export unique film IDs. The film name was the only join key — fragile and requiring manual cleanup.

👥

Multi-person logic

Most SQL tutorials cover one table. Cross-person analysis required UNION ALL, LEFT JOIN with IS NULL, and careful thinking about what "both watched" means in query terms.

Approach

A deliberate skill progression

Rather than jumping straight to complex queries, I built up in levels — each one unlocking the next set of questions I could ask.

1

SELECT, WHERE, ORDER BY

Simple queries to understand the data's shape — what years are covered, how dates are formatted, whether the import worked correctly.

2

GROUP BY, COUNT, AVG, ROUND

Aggregating by year to find average ratings over time and total films watched per period. Surfaced early patterns in watching habits.

3

JOINs — single person

Joined diary, reviews, and watchlist tables. Used LEFT JOIN with IS NULL to find films watched but never reviewed.

4

JOINs — cross-person comparison

Joined my diary against Nirjan's to find shared watches, rating gaps, biggest disagreements, and films only one of us has seen.

Key queries

SQL — Biggest rating disagreements
SELECT d."Name",
       d."Rating"  AS my_rating,
       n.rating    AS nirjan_rating
FROM diary d
JOIN nirjan_diary n ON d."Name" = n.name
ORDER BY ABS(d."Rating" - n.rating) DESC;
SQL — Films Nirjan watched that I haven't
-- A built-in recommendation engine
SELECT n.name
FROM nirjan_diary n
LEFT JOIN diary d ON n.name = d."Name"
WHERE d."Name" IS NULL;
SQL — Who watched more in 2024
SELECT 'me'     AS whose, COUNT(*) AS films_watched
FROM diary
WHERE EXTRACT(YEAR FROM "Date") = 2024
UNION ALL
SELECT 'nirjan', COUNT(*)
FROM nirjan_diary
WHERE EXTRACT(YEAR FROM date) = 2024;
Key findings

What the data actually said

⭐

We agree more than we argue

Most shared watches have ratings within half a star. The genuine disagreements are fewer than expected — but more dramatic when they exist.

📅

Nirjan watches more, I rate higher

Nirjan logs more films per year and tends to be the more critical rater. My average rating across shared watches runs higher — I'm more generous, he's harder to impress.

🎬

Disagreements cluster on prestige films

The biggest rating gaps aren't on obscure picks — they're the films everyone has an opinion on. The canon is contested.

📋

His diary is my watchlist

A large share of films I've saved to watch, Nirjan has already seen — making his diary a better filter than the algorithm.

What I learned

Beyond the syntax

01

Schema issues are real, not academic

Capitalised vs. lowercase column names is exactly the kind of thing that breaks production queries. Working through it manually made the lesson stick in a way no tutorial ever did.

02

The join key matters enormously

Using film names as a join key works until it doesn't — one trailing space, one alternate title, and the match fails silently. The fix is proper IDs, which is what the TMDB enrichment phase will address.

03

Good questions lead to better queries

Starting with "who has better taste?" forced me to define what that actually means in SQL — rating averages, disagreement magnitude, overlap percentage. Translating a real question into a query is a different skill from knowing syntax.
